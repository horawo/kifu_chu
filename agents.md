# AGENTS.md

このリポジトリで Codex が作業する際のルールを定義します。

## 基本方針

- 回答・作業内容は簡潔にする。
- 不要な説明、長い前置き、過剰なログ出力は避ける。
- トークン消費量を抑えるため、必要なファイル・必要な範囲だけを確認する。
- 既存コードの設計・命名・フォーマットにできるだけ合わせる。
- 大きな変更よりも、小さく安全な変更を優先する。
- 推測で作業せず、不明点が作業結果に影響する場合は確認する。

## ファイル操作

- 文字コードは必ず UTF-8、BOM なしで保存する。
- 改行コードは既存ファイルに合わせる。
- 不要なファイルの作成・変更・削除は行わない。
- 自動生成ファイル、ビルド成果物、依存パッケージ本体は原則として編集しない。
- 変更前に、対象ファイルの内容と周辺コードを確認する。

## コーディングルール

- メソッド、関数、クラスには必ずドキュメント用コメントを入れる。
- 既存のコメント形式がある場合はそれに合わせる。
- コメントは「何をするか」だけでなく、必要に応じて「なぜそうするか」も書く。
- 明らかな処理を説明するだけの冗長なコメントは避ける。
- 型情報を使える言語では、可能な限り型を明示する。
- 例外処理・エラー処理を適切に入れる。
- セキュリティ上危険な実装を避ける。
- ハードコードされた秘密情報、API キー、パスワード、トークンを追加しない。
- 既存の公開 API や外部仕様を変更する場合は、影響範囲を明示する。

## 依存関係・コマンド実行

以下のコマンドは、実行前に必ずユーザーへ確認する。

- `npm install`
- `npm add`
- `npm update`
- `npm audit fix`
- `npx`
- `pip install`
- `pipenv install`
- `poetry add`
- `composer install`
- `composer require`
- `composer update`

依存パッケージのインストールや更新は自動で行わない。

必要な場合は、以下を提示してから確認を取る。

- 実行したいコマンド
- 実行理由
- 変更される可能性があるファイル
- 想定されるリスク

例：

```bash
npm install
````

のようなコマンドは、確認なしで実行しない。

## シンタックスチェック・検証

コード変更後は、必ず可能な範囲でシンタックスチェックを行う。

言語別の例：

```bash
node --check path/to/file.js
```

```bash
python -m py_compile path/to/file.py
```

```bash
php -l path/to/file.php
```

```bash
ruby -c path/to/file.rb
```

```bash
tsc --noEmit
```

```bash
composer validate
```

ただし、依存関係の追加・更新が必要な検証コマンドは勝手に実行しない。

シンタックスチェックが実行できない場合は、理由を説明する。

## テスト

* 既存のテストがある場合は、関連するテストを優先して実行する。
* 全体テストは時間や影響が大きい場合があるため、必要に応じて確認する。
* テストが存在しない場合でも、最低限の静的確認・シンタックスチェックを行う。
* テスト失敗時は、原因が今回の変更か既存問題かを切り分ける。

## Git 操作

* `git status` で作業ツリーの状態を確認してから変更する。
* ユーザーが行った変更を上書きしない。
* コミット、プッシュ、リセット、リベース、チェックアウトなどの Git 操作は、明示的に依頼された場合のみ行う。
* 特に以下は確認なしで実行しない。

```bash
git reset
git checkout
git restore
git clean
git rebase
git push
```

## トークン節約のための作業方針

* まず関連しそうなファイル名・ディレクトリ構成を確認する。
* ファイル全体を読む必要がない場合は、必要な範囲だけ読む。
* 同じファイルを何度も読み直さない。
* 大量のログやビルド結果をそのまま出力しない。
* 変更内容の説明は要点だけにまとめる。
* 最終報告には以下だけを簡潔に含める。

  * 変更した内容
  * 実行したチェック
  * 残っている注意点

## 実装前の確認が必要なケース

以下の場合は、実装前にユーザーへ確認する。

* 仕様が曖昧で複数の実装方針があり得る場合
* DB スキーマやマイグレーションに関わる場合
* 外部 API の仕様変更が必要な場合
* 依存パッケージの追加・更新が必要な場合
* 既存の挙動を変える可能性が高い場合
* セキュリティや認証に関わる場合
* 大量のファイル変更が必要な場合

## 実装スタイル

* 小さな単位で変更する。
* 既存の命名規則に合わせる。
* 重複コードは可能な範囲で整理する。
* ただし、過度なリファクタリングは避ける。
* 今回の依頼と直接関係ない改善は、勝手に実施せず提案に留める。
* 既存の lint / formatter 設定がある場合はそれに従う。
* 設定がない場合は、既存コードのスタイルを優先する。

## ドキュメント

* コードの使い方が変わる場合は README や関連ドキュメントの更新を検討する。
* ただし、ドキュメント変更が不要な小修正では無理に更新しない。
* コメント・ドキュメントは日本語または既存ファイルの言語に合わせる。

## セキュリティ

* 秘密情報をログ出力しない。
* `.env` や認証情報ファイルを不用意に表示・編集しない。
* SQL インジェクション、XSS、CSRF、パストラバーサルなどに注意する。
* 外部入力は検証・サニタイズする。
* ファイル削除・上書き・ネットワークアクセス・外部コマンド実行は慎重に扱う。

## 最終報告フォーマット

作業完了時は、以下の形式で簡潔に報告する。

```md
## 変更内容

- ...

## 確認内容

- ...

## 注意点

- ...
```

不要な詳細ログは含めない。
